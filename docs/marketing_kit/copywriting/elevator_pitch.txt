Слайд 01 · Титул
⏱ 0:00–0:30    ·    🎤 PM
	[ На екрані титульний слайд: лого Relativa, слоган, граф-вузли праворуч. Команда стоїть разом. ]
Катерина (PM): Доброго дня, шановна комісіє. Ми — команда Relativa. Сьогодні ми покажемо, як перетворити хаос клієнтських таблиць на чітку картину бізнесу. Наш слоган простий: «Перетворюємо зв'язки на дохід». Relativa показує всю компанію на одному графі відносин, підсвіченому ML-ризиком — замість нескінченних таблиць і звітів. За наступні десять хвилин ми проведемо вас від проблеми до архітектури, а далі покажемо продукт у дії.
Слайд 02 · Проблема
⏱ 0:30–1:30    ·    🎤 PM
	[ Слайд із трьома метрик-картками болю. ]
Катерина (PM): Почнемо з болю, який ми вирішуємо. Уявіть B2B-компанію малого чи середнього розміру. Менеджери тонуть у таблицях, інформація про клієнтів, співробітників та угоди розкидана по різних інструментах. Наслідки — три. Перше: години ручного аналізу замість роботи з клієнтами; за дослідженнями, відділи продажів витрачають до тридцяти відсотків часу просто на консолідацію даних. Друге: майже нульова видимість зв'язків — хто з ким працює, який клієнт критичний, видно лише інтуїтивно. Третє, найдорожче: реакцію на відтік клієнта команда бачить запізно, коли клієнт уже йде. У таблицях структуроване управління зв'язками стає просто неможливим.
Слайд 03 · Рішення / Killer-фіча
⏱ 1:30–2:30    ·    🎤 PM
	[ Синій слайд: живий граф ліворуч (пульсуючий червоний вузол), легенда кольорів праворуч. ]
Катерина (PM): Наша відповідь — і це серце продукту — Relationship Intelligence Graph. Уся компанія на одному інтерактивному графі. Кожен вузол — клієнт, менеджер чи угода — підсвічений ML-ризиком: зелений означає здорові відносини, жовтий — потребує уваги, червоний — високий ризик. Найцінніше тут — самі зв'язки. Ви одразу бачите клієнта, з яким працюють два менеджери; менеджера, чий весь портфель під загрозою; угоди, що групуються навколо одного ключового акаунта. Це форма вашого бізнесу з першого погляду — без жодного звіту.
Слайд 04 · Рішення / MVP
⏱ 2:30–3:30    ·    🎤 PM → Backend
	[ Сітка 3×2 з шести модулів, виділена картка «Граф» з бейджем KILLER. ]
Катерина (PM): Граф — це вершина, але під ним стоїть ядро з шести модулів: управління сутностями, сам граф відносин як killer-фіча, ML-скоринг ризиків, рольовий контроль доступу, динамічна фільтрація та аналітичний дашборд. Ми свідомо окреслили межі MVP: імпорт-експорт даних, ШІ-агенти та зовнішні інтеграції — поза скоупом цього релізу, щоб сфокусуватися на головному. А як саме працює інтелект продукту — передаю слово Назару.

Головна теза для комісії — це наш власний інтелект, а не обгортка над чужим сервісом.

Назар (Backend): Тепер — стек, шар за шаром. Backend стисло проходить по технологіях кожного шару: ........................ ⏱ 5:30–6:45    ·    🎤 Backend + Frontend + DB
	[ Схема: Vue SPA :3000 → Gateway YARP :8080 → сервіси Auth/Core/Graph/ML/Audit → RabbitMQ → PostgreSQL :5432. Троє спікерів ведуть схему згори вниз. ]
Назар (Backend): Архітектурно це мікросервіси. Backend пояснює загальну логіку та серверні рішення: ........................ (єдина точка входу через шлюз YARP; розділення на сервіси Auth, Core, Graph, ML, Audit зі своїми портами; чому обрано мікросервіси замість моноліту; інженерні виклики — транзакційний Outbox для гарантованого аудиту, гнучка EAV-модель без міграцій схеми, дворівневий RBAC). Пройдемося по схемі згори вниз — почнемо з того, як до цього всього звертається клієнт. Наталіє.
Наталія (Frontend): Frontend пояснює верхній шар взаємодії: ........................ (Vue 3 SPA як єдиний клієнт; усі запити йдуть через один шлюз, а не напряму до сервісів; JWT-автентифікація на боці клієнта; real-time оновлення графа через SignalR — інтерфейс не треба перезавантажувати, дані прилітають самі). Тобто для користувача вся ця складність схована за однією точкою входу. А що відбувається глибше, на рівні обміну та зберігання даних — краще розкаже Влад.
Влад Антонюк (DB): DB пояснює нижній шар — обмін подіями та сховище: ........................ (асинхронний обмін доменними й аудит-подіями через RabbitMQ; чому саме черга, а не прямі виклики між сервісами; PostgreSQL як єдине джерело правди; роль Outbox у консистентності даних). А тепер детальніше про те, як саме влаштовані самі дані — переходимо до структури.
